## 📝 2. 내용 정리

### 쿠버네티스의 역할
- 서로 다른 서버에서 작동하는 컨테이너를 한번에 관리
  - 만약 도커라면, 같은 명령어를 100번은 쳤어야 할 거임

### 쿠버네티스 구조
- 마스터, 워커 노드로 구분
- CNI (Container Network Interface) : 클러스터 내의 컨테이너 간의 통신을 위한 인터페이스
- 컨트롤 플레인 : 쿠버네티스 클러스터 전반의 작업을 관리 (API 서버, etcd, 스케줄러, 컨트롤러 매니저)
- API 서버 : kubectl 명령어로 마스터 노드의 API 서버에 API요청을 보내면서 이루어진다. (프론트엔드 같은 역할)
- etcd : 쿠버네티스 클러스터에 존재하는 모든 데이터를 저장하는 key-value 저장소
- 스케줄러 : 파드를 어느 노드에서 실행해야 할지 결정하는 역할
- 컨트롤러 매니저 : 쿠버네티스 리소스를 관리하고 제어하는 역할, 마스터에서 실행되며 상태를 모니터링 한다.
  - 디플로어먼트 컨트롤러, 서비스 컨트롤러,, 레플리카셋 컨트롤러가 존재한다.

### 노드
- kublet : kublet 파드 내부의 컨테이너 실행을 담당한다.
  - 상태를 모니터링하고 문제가 있으면 재배포도 담당한다.
- kube-proxy : 노드에서 네트워크 역할을 수행한다.
- 컨테이너 런타임 : 컨테이너 생명주기를 담당한다.
- 파드 : 컨테이너를 그룹화한 형태. 쿠버네티스의 최소 단위. 파드는 실행할 때마다 IP주소가 변경된다.

#### 워커 노드
- 실행중인 컨테이너의 집합
- 레플리카셋 : 파드의 복제 관리, 모니터링을 담당
- 디플로이먼트 : 배포와 스케일링을 담당한다.
- 스테이트풀셋 : 파드 사이에서 순서와 고유성이 보장되는 경우에 사용한다.
- 데몬셋 : 쿠버네티스를 구성하는 모든든 노드가 파드의 복사본을 실행시킴. 
- 잡과 크론잡 : 작업이 정상적으로 완료되고 종료된느 것을 담당

### 네트워크
- 서비스 : 서비스를 사용해 파드를 여러개로 묶어 클러스터 외부로 노출시킨다. (내부를 수정할 필요 없음)
- 인그레스 : 쿠버네티스 내부의 서비스를 HTTP/HTTPS 클러스터 외부로 라우팅 한다.

### 스토리지
- 파드의 상태와 상관없이 파일을 보관
